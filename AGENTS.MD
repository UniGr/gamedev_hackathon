# AGENTS.MD — Session Integration Contract

## Цель этого файла
Этот документ фиксирует правила разработки **в рамках текущей сессии** так, чтобы изменения:
- были максимально безопасными для существующего кода;
- легко мерджились в основную ветку;
- добавлялись преимущественно через **новые изолированные модули**.

---

## 1) Главный принцип: минимальное повреждение исходных файлов

1. Существующие файлы считаются «стабильным ядром».
2. Любые новые механики добавляются в первую очередь через **новые файлы/сцены/скрипты**.
3. Изменения в старых файлах разрешены только как:
   - точечное подключение нового модуля;
   - регистрация сигнала;
   - добавление узла-инициализатора.
4. Запрещено:
   - массовое форматирование;
   - переименование старых публичных сущностей без крайней необходимости;
   - рефакторинг «ради красоты» в исторических файлах.

---

## 2) Merge-friendly правила (обязательно)

1. Один смысловой блок = один отдельный модуль/файл.
2. Новая логика должна быть декомпозирована:
   - поведение сущности;
   - спавн;
   - конфиг баланса;
   - эффекты (VFX/SFX) — отдельные сущности.
3. В старые файлы вносим только «точки врезки» (thin integration layer).
4. Любой новый модуль должен иметь:
   - краткий комментарий назначения;
   - минимальный публичный API;
   - отсутствие жёстких прямых зависимостей на UI.
5. Для каждой новой фичи — отдельный список изменённых файлов и назначение.

---

## 3) План фич текущей сессии (без ломки старого)

### 3.1 Красные налётчики (Raiders)

**Цель:** добавить врагов, которые подлетают к кораблю и «выгрызают» (удаляют/повреждают) модули.

**Требования к реализации:**
1. Новый модуль сущности налётчика (отдельный скрипт + сцена).
2. Новый спавнер налётчиков (отдельный скрипт + сцена).
3. Логика выбора цели:
   - приоритет ближайшего доступного внешнего модуля;
   - fallback на любой доступный модуль при отсутствии внешнего.
4. Логика атаки:
   - подлёт -> захват цели -> таймер «грызения» -> нанесение урона/удаление.
5. События через Event Bus (по возможности), без жёстких ссылок между системами.

### 3.2 Небольшой боковой дрейф корабля

**Цель:** добавить лёгкое визуальное «живое» движение корабля влево-вправо.

**Требования:**
1. Отдельный модуль sway/drift (скрипт-компонент).
2. Параметры амплитуды/частоты вынести в exports/config.
3. Без влияния на логику сетки и коллизии модулей (только визуальный слой).

### 3.3 Временные визуальные эффекты (VFX)

**Цель:** добавить временные эффекты для прототипирования боя.

**Минимум:**
1. Эффект попадания налётчика по модулю.
2. Эффект разрушения/отгрызания модуля.
3. Эффект спавна налётчика.

### 3.4 Временные звуковые эффекты (SFX)

**Цель:** добавить простые временные звуки для обратной связи.

**Минимум:**
1. Звук появления налётчика.
2. Звук атаки/грызения.
3. Звук уничтожения/откуса модуля.

> Все SFX/VFX считаются временными (placeholder) и должны быть легко заменяемыми.

---

## 4) Маленькая балансировка (микро-баланс)

Баланс в этой сессии делается минимальным и безопасным:
1. Налётчик:
   - умеренная скорость подлёта;
   - заметная задержка перед «отгрызанием»;
   - ограничение одновременного числа налётчиков.
2. Спавн:
   - плавный рост частоты со временем (без резких пиков);
   - верхний лимит активных налётчиков на экране.
3. Защита игрока:
   - оставить окно реакции (налётчик не должен мгновенно удалять модуль);
   - параметры должны быть вынесены в настраиваемые поля.

---

## 5) Техническая стратегия изменений

### 5.1 Что создаём новыми файлами
- `raider` сущность
- `raider_spawner`
- `raider_ai` (если нужен отдельный слой)
- `raider_config` (баланс-конфиг)
- `raider_vfx` / `raider_sfx` помощники
- `ship_sway` (визуальный дрейф)

### 5.2 Что можно слегка тронуть в старых файлах
Только:
1. Подключение новых модулей в главной сцене/цикле.
2. Подписка на новые сигналы.
3. Передача безопасных ссылок/колбеков для удаления модулей.

---

## 6) Ограничения и гарантия совместимости

1. Не ломать существующую механику строительства.
2. Не менять контракты старых модулей без необходимости.
3. Избегать cross-module циклических зависимостей.
4. Каждое изменение должно быть потенциально отключаемым (feature toggle или удалением узла/сцены).

---

## 7) Definition of Done для этой сессии

Фича считается завершённой, если:
1. Налётчики спавнятся, подлетают и могут выгрызать модули.
2. Корабль имеет лёгкий визуальный дрейф в стороны.
3. Добавлены временные VFX и SFX для ключевых событий.
4. Основные старые файлы не подверглись крупным изменениям.
5. Изменения структурированы по отдельным модулям и готовы к аккуратному merge.

---

## 8) Формат дальнейших коммитов (рекомендуется)

- `feat(raiders): add raider entity and attack flow`
- `feat(raiders): add modular spawner and balance config`
- `feat(vfx-sfx): add temporary combat feedback effects`
- `feat(ship): add lightweight visual side drift`
- `chore(integration): wire modules with minimal legacy touch`

---

## 9) Приоритет при конфликте требований

Если есть конфликт между «быстро сделать» и «безопасно для мерджа», всегда выбирается:
1. модульность,
2. минимальные правки старого,
3. предсказуемая интеграция.
